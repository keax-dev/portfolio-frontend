# Principios obligatorios

Aplica los siguientes principios durante toda la revisión:

1. Organiza el proyecto por funcionalidades o dominios de negocio, preferiblemente con arquitectura feature-first.

2. Utiliza componentes standalone, ya que el proyecto está construido con Angular 22.

3. Utiliza lazy loading para funcionalidades, páginas y secciones que no necesitan cargarse inicialmente.

4. Utiliza Signals para estado síncrono, local, compartido o derivado cuando sea apropiado.

5. Utiliza `computed()` para información derivada y evita almacenar datos que puedan calcularse a partir de otra fuente.

6. Utiliza `effect()` solamente para efectos secundarios reales. No utilices `effect()` para sincronizar estado que pueda resolverse mediante `computed()`, eventos o un diseño más claro.

7. Utiliza RxJS para flujos asíncronos, cancelación, debounce, solicitudes dependientes, combinación de observables, eventos en el tiempo, WebSockets o concurrencia.

8. No reemplaces indiscriminadamente RxJS con Signals.

9. Utiliza `input()`, `output()` y `model()` cuando aporten claridad y sean adecuados para el componente.

10. Utiliza `inject()` para la inyección de dependencias cuando mejore la legibilidad.

11. Utiliza el control flow nativo de Angular:

* `@if`
* `@for`
* `@switch`
* `@empty`

12. Utiliza `@defer` solamente para contenido pesado, secundario o no crítico.

13. Utiliza interceptores funcionales.

14. Utiliza guards funcionales cuando correspondan.

15. Utiliza `takeUntilDestroyed()` para suscripciones imperativas legítimas.

16. Utiliza `httpResource()` o Resource API solamente cuando sean estables, compatibles y representen una mejora real frente a HttpClient y RxJS.

17. No migres formularios existentes únicamente por modernización estética.

18. No introduzcas NgRx, Akita, Elf u otra librería de estado salvo que la complejidad real del proyecto lo justifique claramente.

19. No introduzcas patrones, capas o abstracciones que solo aumenten la cantidad de archivos.

20. Toda modificación debe mejorar de forma verificable al menos uno de estos aspectos:

* Mantenibilidad.
* Tipado.
* Rendimiento.
* Testabilidad.
* Seguridad.
* Claridad.
* Escalabilidad.
* Reducción de duplicación.
* Reducción de acoplamiento.

# Core

La carpeta `core` debe contener únicamente elementos globales y de una sola instancia, por ejemplo:

* Autenticación.
* Configuración de aplicación.
* Layout principal.
* Interceptores globales.
* Manejo global de errores.
* Servicios globales reales.
* Inicialización de la aplicación.

No coloques en `core` elementos específicos de una funcionalidad.

# Shared

La carpeta `shared` debe contener únicamente elementos:

* Genéricos.
* Reutilizables.
* Independientes de un dominio.
* Sin lógica específica del negocio.

No conviertas `shared` en un contenedor general para cualquier archivo utilizado más de una vez.

Un componente utilizado por dos pantallas del mismo dominio puede permanecer dentro de su propia feature.

# Estado

Analiza cuidadosamente dónde debe vivir cada dato.

Clasifica el estado como:

* Estado local del componente.
* Estado derivado.
* Estado de formulario.
* Estado compartido dentro de una feature.
* Estado global.
* Estado persistido.
* Estado representado en la URL.
* Estado remoto proveniente del backend.

Aplica las siguientes reglas:

* El estado visual simple debe permanecer local.
* El estado derivado debe implementarse con `computed()`.
* El estado compartido por una feature puede gestionarse mediante un servicio con Signals.
* El estado global debe reservarse para información verdaderamente global.
* Los filtros, paginación, ordenamiento o selección que deban poder compartirse o restaurarse pueden representarse en query params.
* No dupliques una misma fuente de verdad.
* No almacenes globalmente información utilizada por una sola pantalla.
* No crees stores gigantes.
* No combines estados de dominios no relacionados.
* No almacenes manualmente datos que puedan derivarse.

Cuando sea apropiado, proporciona un store de feature en el nivel de ruta o componente para que su ciclo de vida quede limitado a esa funcionalidad.

Ejemplo conceptual:

```ts
@Injectable()
export class ExampleStore {
  private readonly api = inject(ExampleApi);

  private readonly itemsState = signal<Example[]>([]);
  private readonly loadingState = signal(false);
  private readonly errorState = signal<string | null>(null);

  readonly items = this.itemsState.asReadonly();
  readonly loading = this.loadingState.asReadonly();
  readonly error = this.errorState.asReadonly();

  readonly activeItems = computed(() =>
    this.items().filter(item => item.active),
  );
}
```

No copies este patrón de forma automática. Adáptalo según las necesidades reales.

# Signals

Utiliza Signals para:

* Estado local.
* Estado síncrono.
* Estado derivado.
* Selección actual.
* Estado visual.
* Información compartida de una feature.
* Datos consumidos frecuentemente desde templates.

Evita:

* Mutaciones directas.
* Duplicación de estado.
* Signals públicos modificables cuando deberían ser de solo lectura.
* `effect()` utilizado para copiar un Signal a otro.
* Crear Signals innecesarios para valores constantes.
* Convertir todos los Observables a Signals sin una razón.
* Crear `toSignal()` repetidamente dentro de getters o métodos.
* Usar Signals como sustitución forzada de RxJS.

# RxJS

Revisa:

* `subscribe()` anidados.
* Memory leaks.
* Observables que nunca se completan.
* Suscripciones duplicadas.
* Solicitudes HTTP repetidas.
* Falta de cancelación.
* Ausencia de debounce.
* Manejo incorrecto de errores.
* Uso incorrecto de operadores de concurrencia.
* `subscribe()` utilizado solamente para copiar valores.
* Uso innecesario de Subjects.
* Subjects expuestos públicamente.
* Conversión innecesaria entre Signals y Observables.

Selecciona correctamente:

* `switchMap` para cancelar solicitudes anteriores.
* `concatMap` para respetar orden secuencial.
* `mergeMap` para concurrencia.
* `exhaustMap` para ignorar nuevas emisiones mientras una operación está activa.
* `forkJoin` para operaciones que deben completarse juntas.
* `combineLatest` para combinar fuentes activas.
* `debounceTime` y `distinctUntilChanged` para búsquedas.
* `shareReplay` únicamente cuando exista una necesidad clara de compartir resultados.

No utilices `shareReplay(1)` indiscriminadamente.

Cuando exista una suscripción imperativa legítima, utiliza:

```ts
takeUntilDestroyed()
```

No agregues un `Subject<void>` manual para destruir suscripciones cuando Angular 22 ya proporciona una solución adecuada.

# Routing

Comprueba:

* Lazy loading.
* `loadComponent`.

* Providers por ruta.
* Guards.
* Resolvers.
* Redirecciones.

* Parámetros.

Ejemplo:

```ts

    path: 'employees',
    loadChildren: () =>
      import('./features/employees/employees.routes')

  },
];
```

Dentro de una feature:

```ts
export const EMPLOYEE_ROUTES: Routes = [
  {
    path: '',
    providers: [EmployeeStore],
    children: [
      {
        path: '',
        loadComponent: () =>
          import('./pages/employee-list-page')
            .then(component => component.EmployeeListPage),
      },
      {
        path: ':id',
        loadComponent: () =>
          import('./pages/employee-detail-page')
            .then(component => component.EmployeeDetailPage),
      },
    ],
  },
];
```

No conviertas todas las rutas automáticamente.

Analiza primero si el cambio mejora realmente la carga inicial y la separación de la feature.

* Texto alternativo.
* Uso correcto de ARIA.

Utiliza elementos HTML nativos antes de agregar roles ARIA.

Un elemento interactivo debe ser realmente interactivo.

Ejemplo:

```html
<button type="button" (click)="openDetail()">
  Ver detalle
</button>
```

Evita:

```html
<div (click)="openDetail()">
  Ver detalle
</div>
```

# Estilos

Revisa:

* Estilos globales innecesarios.
* Selectores demasiado específicos.
* `!important`.
* Duplicación.
* Valores mágicos.
* Estilos inline.
* Clases ambiguas.
* Dependencias entre estilos de features.
* Responsividad.
* Consistencia visual.
* Variables o tokens de diseño.

No cambies el diseño actual salvo que sea necesario para:

* Corregir un error.
* Mejorar accesibilidad.
* Solucionar responsividad.
* Eliminar inconsistencias evidentes.

No reemplaces toda la estrategia CSS sin una necesidad real.

# Testing

Revisa la configuración actual de testing.

Utiliza la herramienta ya configurada en el proyecto.

Si el proyecto utiliza Vitest, continúa con Vitest.

Si utiliza Jasmine y Karma, no migres automáticamente sin evaluar costo, beneficio y riesgo.

Agrega pruebas útiles para:

* Reglas de negocio.
* Mappers.
* Validadores.
* Stores.
* Servicios críticos.
* Interceptores.
* Guards.
* Componentes con interacción.
* Formularios.
* Flujos críticos.

No escribas pruebas que solamente verifiquen:

```ts
expect(component).toBeTruthy();
```

Prioriza pruebas de comportamiento.

Ejemplos:

* Dado un DTO, el mapper genera correctamente el modelo.
* Dado un filtro, el store devuelve los registros esperados.
* Dado un error del backend, el formulario muestra el mensaje correcto.
* Al hacer clic, el componente emite el evento esperado.
* Una solicitud previa se cancela al cambiar la búsqueda.
* Un usuario sin permiso no puede ejecutar una acción.
* Un formulario inválido no envía información.

No busques aumentar cobertura con pruebas sin valor.

* `console.error`.
* Logs sensibles.
* Logs duplicados.
* Información de tokens.
* Información personal.
* Logs activos en producción.

# Dependencias

* Dependencias duplicadas.
* Librerías pesadas.

# Calidad y consistencia

* Convenciones de nombres.
* Nombres de componentes.
* Nombres de selectors.
* Nombres de rutas.
* Nombres de servicios.
* Nombres de modelos.

* TODO sin contexto.

# Reglas para evitar sobreingeniería

No debes:

* Aplicar Clean Architecture completa a cada CRUD.
* Crear interfaces para todas las clases sin necesidad.
* Crear repositories que solo deleguen una llamada sin aportar aislamiento.
* Crear facades que solo reexporten métodos.
* Crear use cases de una línea.
* Crear stores globales innecesarios.
* Crear librerías internas para código usado una sola vez.
* Dividir un componente pequeño en muchos archivos.
* Introducir patrones únicamente para aparentar seniority.
* Crear abstracciones antes de que exista duplicación o variación real.
* Generalizar componentes de dominios diferentes solo porque visualmente se parecen.
* Agregar NgRx sin justificación.
* Reemplazar RxJS con Signals de manera indiscriminada.
* Migrar APIs funcionales que ya son correctas únicamente por moda.
* Renombrar masivamente archivos sin beneficio.
* Cambiar el diseño visual.
* Cambiar contratos del backend.
* Alterar reglas de negocio sin evidencia.
* Silenciar errores de compilación.
* Desactivar reglas de lint para evitar corregir problemas.
* Utilizar `skipLibCheck` como solución rápida.
* Utilizar `any`.
* Utilizar `npm audit fix --force`.
